# System Technical Documentation

## Abstract

This project provides a complete real-time sensor pipeline built with Raspberry Pi 5, ADXL345, MQTT, and a Qt desktop client. It covers data acquisition, network transmission, visualization, and local actuator linkage in one flow.

## 1. Architecture

The system has four layers:

1. Sensing layer: ADXL345 over I2C on Raspberry Pi
2. Transport layer: MQTT (Mosquitto broker + Paho C client)
3. Visualization layer: Qt Widgets desktop app
4. Actuator layer: LED linkage on Raspberry Pi

## 2. Data Flow

1. Raspberry Pi reads ADXL345 X/Y/Z registers
2. Device computes pitch/roll
3. Device adds CPU metrics and timestamp
4. Device publishes JSON to MQTT topic
5. Qt app subscribes, parses, logs, plots, and computes rolling stats

## 3. Core Source Files

### Qt side

- `src/main.cpp`: app entry
- `include/mainwindow.h`: main window declarations
- `src/mainwindow.cpp`: MQTT logic, parsing, plotting, stats
- `EEN1071Ass2.pro`: qmake project config
- `tools/build_qt.ps1`: build script
- `tools/run_qt.ps1`: run script
- `tools/package_qt.ps1`: packaging script

### Raspberry Pi side

- `raspberry_pi/程序源码/ADXL345.h/.cpp`: sensor driver and angle computation
- `raspberry_pi/程序源码/I2CDevice.h/.cpp`: I2C wrapper
- `raspberry_pi/程序源码/publisher.cpp`: MQTT publisher
- `raspberry_pi/程序源码/subscriber_led_wiringpi.cpp`: LED linkage subscriber

## 4. MQTT Topics and Protocol

Default topics:

- Data topic: `sensor/adxl345`
- Status topic: `sensor/adxl345/status`

Status logic:

- Publish retained `online` after successful connect
- Configure retained Last Will `offline` for unexpected disconnect

QoS:

- Publisher accepts QoS argument `0/1/2`
- Qt supports independent subscribe/publish QoS selection

## 5. JSON Schema

Example:

```json
{
  "pitch": -31.88,
  "roll": -52.19,
  "accel_x": -129,
  "accel_y": -193,
  "accel_z": -76,
  "cpu_load": 0.12,
  "cpu_temp_c": 51.8,
  "timestamp": "2026-04-10T07:17:46Z"
}
```

Main fields:

- `pitch`, `roll`: orientation angles (deg)
- `accel_x`, `accel_y`, `accel_z`: raw acceleration values
- `cpu_load`: system load
- `cpu_temp_c`: CPU temperature in Celsius
- `timestamp`: UTC time (ISO8601)

## 6. Deployment Steps

1. Start Mosquitto broker and verify connectivity
2. Build and run Qt client
3. Build and run Raspberry Pi publisher (`./publisher 0|1|2`)
4. Confirm live logs/charts/stats in Qt
5. Optionally run LED subscriber for actuator demonstration

## 7. Validation Checklist

- MQTT connectivity is stable
- Qt receives and parses JSON correctly
- Real-time curve updates and 60-second stats are correct
- Status topic reflects online/offline transitions
- LED level switches with pitch thresholds

## 8. Troubleshooting

- No data in Qt: check topic alignment and JSON field names
- Sensor read error: check I2C enablement, wiring, and address `0x53`
- LED no response: check WiringPi install and GPIO mapping
- Broker connect issues: verify host, port, credentials, and broker service status

## 9. Conclusion

The project demonstrates a practical, end-to-end IoT pipeline from embedded sensing to desktop visualization, with clear extensibility for additional sensors, multi-topic expansion, and long-term storage integration.
